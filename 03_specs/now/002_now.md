## Outcome
Crear una skill reusable llamada `founder-cto` para actuar como sistema CTO
agentic general en proyectos del fundador: preventa tecnica, decisiones de
arquitectura, build-vs-buy, revision de specs, readiness de ejecucion, weekly
CTO, postmortems y gobierno de riesgo/deuda tecnica.

La skill debe ser generica y portable, no especifica de EPI10. Debe vivir en el
repo como fuente controlada y dejar un paquete exportable para instalarla
globalmente mas adelante.

## Required Skill Direction

### `SKILL.md`
- Frontmatter:
  - `name: founder-cto`
  - `description`: pushy enough to trigger whenever the user asks for CTO
    judgment, architecture decisions, spec review, build-vs-buy, technical
    pre-sales, delivery governance, risk/debt review, project planning, or
    weekly CTO-style review.
- Body must be concise and route detail to references.
- Body must include:
  - overview;
  - default output contract: respond in Spanish unless asked otherwise;
  - core workflow:
    1. classify project/context;
    2. choose rigor dial;
    3. select mode;
    4. identify decisions, risks, assumptions, unknowns;
    5. produce a concrete governance artifact;
  - rule that the skill is advisory/governance by default, not an execution
    worker;
  - rule to avoid overengineering growth/sales/internal hacks while raising
    rigor for client delivery and sensitive projects;
  - links to all reference files with clear "read when..." guidance.

### `references/modes.md`
- Define each mode, when to use it, required inputs, and expected output shape.
- Make the weekly CTO mode mimic a 30-40 minute CTO sparring session:
  progress, decisions made, blocked decisions, risks, next decisions, and
  escalations.

### `references/decision-gates.md`
- Define gate checks for build-vs-buy, architecture, data/security/privacy,
  maintainability, delivery readiness, observability/support, and handoff.
- Include the rigor dial with explicit consequences:
  - low friction for `sales/growth asset`;
  - moderate traceability for `internal ops`;
  - high maintainability/testing/docs for `client delivery`;
  - privacy/security/legal/human-review bias for `regulated/sensitive`.

### `references/output-templates.md`
- Provide compact Markdown templates for all required artifacts.
- Templates should be easy to paste into specs, ADRs, review comments, or
  executor prompts.

### `references/project-adaptation.md`
- Explain how to adapt the CTO skill to any project:
  read repo instructions, current context, active spec, decisions/backlog,
  architecture docs, tests/build commands, and constraints before advising.
- Include a rule that project-specific facts must come from the current repo or
  user-provided material, not from the reusable skill.

### `references/source-notes.md`
- Summarize external sources in short, attributed notes.
- Do not paste long source text.
- Capture only transferable ideas:
  - CTO agent as technical leadership/governance;
  - forcing questions and specialist routing;
  - system design flow: use cases, constraints, high-level design, core
    components, scale/tradeoffs;
  - "best code is code not written" / YAGNI / reuse-before-build bias.

### `evals/evals.json`
- Use the schema style from `shared/skills/skill-creator/references/schemas.md`.
- Include at least 4 evals.
- Each eval must have:
  - `id`
  - `prompt`
  - `expected_output`
  - `files` as an empty list unless a real fixture is created
  - `expectations` with objective checks

## Risks
- The skill could become too broad and generic to be useful. Keep the workflow
  tight: classify, choose rigor, select mode, surface decisions/risks, output a
  concrete artifact.
- The skill could overcorrect and slow down growth/sales work. Preserve low
  friction for `sales/growth asset`.
- The skill could leak project-specific assumptions into reusable guidance. Keep
  project facts in project context, not in the skill.
- External sources may change. Use them as inspiration only and cite them in
  `source-notes.md`; do not depend on exact line numbers or copied content.
