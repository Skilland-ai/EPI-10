## Work Steps

### 1. Build content inventory
Create `supporting/diagram_content_inventory_v1.md`.

The inventory must list each information group that needs to be represented and
where it appears in both new outputs. Use concise tables.

At minimum cover:

- actors: cliente final, Aitor/equipo EPI10, Carmen/direction, Raul, Fer and
  external providers;
- entry point: web/pago with `lead_created` and `payment_success`;
- EPI10 MVP orchestration service in AWS Espana using TypeScript/NestJS;
- PostgreSQL technical DB;
- Healthie portal/API/webhook boundary;
- Odoo operational backoffice;
- EPI10 Informe Final Copilot;
- TellmeGen external/manual boundary;
- Flow 1: lead/payment to operational case;
- Flow 2: portal onboarding to Odoo state;
- Flow 3: TellmeGen external/manual;
- Flow 4: report production and delivery;
- state model draft;
- integration contracts to define;
- data classes and storage boundaries;
- idempotency, retries, dead-letter, alerts and manual recovery;
- observability/support;
- failure modes;
- security/privacy notes;
- solo-developer sequencing;
- ADRs, Fer validation questions, workshop-bound decisions and unresolved
  unknowns.

### 2. Create Excalidraw redesign
Create `architecture_system_design_excalidraw_v1.excalidraw`.

The Excalidraw should be one whiteboard with clear visual zones or frames:

- left: actors and entry events;
- center: EPI10-owned MVP orchestration service, internal modules and
  PostgreSQL technical DB;
- right: SaaS/external systems: Odoo, Healthie, TellmeGen and Copilot;
- bottom or side rail: state model, data boundaries, failure/recovery paths and
  ADR/gate/unknowns panel.

Design requirements:

- Use swimlanes, groups, section titles and a small legend.
- Use consistent color semantics:
  - EPI10-owned components;
  - external/SaaS systems;
  - manual process;
  - sensitive data boundary;
  - unknown/gated decision;
  - error/retry/recovery path.
- Number flows 1 through 4 and keep arrows readable.
- Keep labels short inside shapes; move detail into nearby callouts where
  needed.
- Make sensitive data posture explicit, including:
  - no raw genetic data in Odoo or PostgreSQL;
  - PostgreSQL stores IDs, event metadata, state snapshots, idempotency,
    retries/errors and sanitized audit markers only;
  - Copilot is draft support with pseudonymization and mandatory human review.
- Represent Fer as reviewer of ADRs, risks, state model and slice decisions.

The output must be valid Excalidraw JSON with `type` set to `excalidraw` and
non-empty `elements`.

### 3. Create Mermaid redesign
Create `architecture_system_design_mermaid_v1.md`.

The Mermaid document must be readable as Markdown and can include multiple
Mermaid diagrams if this improves legibility. Required sections:

- overview;
- diagram 1: architecture context using `flowchart`;
- diagram 2: event/data flow using `sequenceDiagram`;
- diagram 3: operational state lifecycle using `stateDiagram-v2`;
- diagram 4: data boundaries, security and recovery paths using `flowchart`
  or another suitable Mermaid diagram.

Each diagram must have a short interpretation paragraph before or after the
fenced Mermaid block. The file must preserve depth without forcing everything
into one unreadable diagram.

### 4. Final QA note
At the bottom of `architecture_system_design_mermaid_v1.md`, add a brief QA
section that states:

- whether the Mermaid version preserves the same architectural information as
  the inventory;
- any information that is intentionally represented as text rather than diagram;
- unresolved unknowns that remain from the source architecture package.

## Interaction Rules
- Ask the user only if a missing fact would change the diagram content.
- Do not ask about style preferences; choose a clean, CTO-review-friendly
  layout.
- If a source contains an unknown, preserve it as `Unknown` or gated.
- If a diagram becomes too dense, split it into more sections/diagrams rather
  than dropping information.
- Be critical about readability, but do not critique or reopen the architecture
  unless a source contradiction blocks representation.

## Risks
- The source design is dense. Mitigation: require a content inventory and allow
  multiple Mermaid diagrams instead of one overloaded view.
- Excalidraw can become text-heavy. Mitigation: use short labels, callouts,
  numbered flows and visual grouping.
- Some Blueprint visual details may not be locally inspectable. Mitigation:
  treat the local Blueprint brief and system design draft as the source of
  truth if the link cannot be opened.
